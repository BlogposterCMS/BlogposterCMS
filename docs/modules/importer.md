# Importer

## Boundaries

Importer is a core-only migration contract. Admin apps call it through Runtime
Manager, and importers write by emitting existing core content, media, theme
and comment events instead of touching storage directly. Local
paths are accepted only from configured import staging roots, and
`runImport.options` cannot override control fields such as `jwt`,
`decodedJWT`, `motherEmitter` or importer-owned payload objects.

Loads content importers from the `importers` subdirectory. Importers can migrate data from other platforms such as WordPress.

## Startup
- Core module.
- Requires a valid JWT.

## Purpose
- Lists available importers via `listImporters`.
- Runs a named importer with `runImport`.
- Provides WordPress WXR dry-run planning before writing content.
- Applies imports through existing core events instead of writing directly to storage.

## Listened Events
- `listImporters`
- `runImport`

Importers are executed inside the server process but must conform to the security rules of the event bus.
Both events require `jwt`, `moduleName: "importer"` and `moduleType: "core"`.

## Permissions

- `importers.list` allows listing installed importers.
- `importers.run` allows running dry-run or write imports.

If `decodedJWT` is omitted, core bootstrap calls can still run. User-facing import flows should always attach a decoded user token.
`runImport` is not a public `/api/meltdown` target; admin/editor clients use
`runtimeManager.cmsAdminApiRequest` with resource `importers` and action `run`.
`runImport.options` is treated as user input. It cannot override control fields
such as `motherEmitter`, `jwt`, `decodedJWT`, `importPayload` or `themeBaseDir`.
Local file and folder paths are accepted only from import staging roots:
`BLOGPOSTER_IMPORT_ROOTS`, `IMPORT_STAGING_DIR`, `temp_uploads/imports`, or
`data/imports`.

## WordPress Importer

The `wordpress` importer accepts one of:

- `options.xml`
- `options.buffer`
- `options.filePath` inside an import staging root

By default it returns a dry-run plan containing site metadata, authors,
WordPress legacy terms, entries, attachments, comments and skipped
WordPress-only item types. Passing `dryRun: false` applies the plan by emitting
`registerContentType`, `createContentEntry`, `createMediaAttachment` and
`createComment`. WordPress categories/tags are preserved under each entry's
`metadata.wordpress.terms` for migration traceability; Blogposter collections
are page parent/child structures, not Content Engine taxonomies.

The parser targets WordPress WXR exports, the XML format generated by WordPress' export tool.

## HTML Theme Importer

The `htmlTheme` importer accepts `options.sourceDir` pointing at a static HTML
theme folder inside an import staging root. Dry-run mode scans the folder,
detects HTML/CSS/image/font assets, extracts a page title and local asset
references, and returns an install plan.

Themes are presentation-only. The importer blocks installation when the source
contains JavaScript files, script tags, inline event handlers, executable source
files, remote references, CSS imports, remote CSS URLs or unsafe CSS behavior.
Those behaviors belong in widgets, modules or apps before the theme can be
installed.

Passing `dryRun: false` installs the theme under `public/themes/<slug>` by
copying the source files into `source/` and generating `theme.json` and
`theme.css`. Existing themes are not replaced unless `overwrite: true` is
passed.

## WordPress Site Package Importer

The `wordpressSitePackage` importer accepts a Blogposter WordPress site package
from one of:

- `options.packageDir`
- `options.sourceDir`
- `options.filePath` or `options.zipPath` pointing at a `.zip`
- `options.manifest` for tests or trusted internal calls

Package files must stay inside an import staging root when called through
`runImport`.

The package manifest must use one of these formats:

- `blogposter-wordpress-site-package`
- `wordpress-site-package`
- `wordpressSitePackage`

The importer keeps WordPress migration responsibilities split:

- `theme.styles` become candidate theme presentation assets.
- `pages[].rendered.htmlPath` or inline rendered HTML become page fallback
  content.
- `content.wxr` can reuse the existing WordPress WXR importer.
- `media[]` entries are applied through Media Manager events.
- `pages[].rendered.scripts` are reported as blocked behavior and are not
  converted into theme assets.
- `theme.scripts` blocks the package before writes begin.

Passing `dryRun: false` applies safe media and rendered page fallbacks through
existing core events. JavaScript behavior still needs to be rebuilt as widgets,
modules or apps.
