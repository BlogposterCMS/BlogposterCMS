# Importer

## Boundaries

Importer is a core-only migration contract. Admin apps call it through Runtime
Manager, and importers write by emitting existing core content, media, theme
and comment events instead of touching storage directly. Local
paths are accepted only from configured import staging roots, and
`runImport.options` cannot override control fields such as `jwt`,
`decodedJWT`, `motherEmitter` or importer-owned payload objects.

Loads content importers from the `importers` subdirectory. Importers can migrate data from other platforms such as WordPress.

## Startup
- Core module.
- Requires a valid JWT.

## Purpose
- Lists available importers via `listImporters`.
- Runs a named importer with `runImport`.
- Provides WordPress WXR dry-run planning before writing content.
- Applies imports through existing core events instead of writing directly to storage.

## Listened Events
- `listImporters`
- `runImport`

Importers are executed inside the server process but must conform to the security rules of the event bus.
Both events require `jwt`, `moduleName: "importer"` and `moduleType: "core"`.

## Permissions

- `importers.list` allows listing installed importers.
- `importers.run` allows running dry-run or write imports.

If `decodedJWT` is omitted, core bootstrap calls can still run. User-facing import flows should always attach a decoded user token.
`runImport` is not a public `/api/meltdown` target; admin/editor clients use
`runtimeManager.cmsAdminApiRequest` with resource `importers` and action `run`.
`runImport.options` is treated as user input. It cannot override control fields
such as `motherEmitter`, `jwt`, `decodedJWT`, `importPayload` or `themeBaseDir`.
Local file and folder paths are accepted only from import staging roots:
`BLOGPOSTER_IMPORT_ROOTS`, `IMPORT_STAGING_DIR`, `temp_uploads/imports`, or
`data/imports`.

## WordPress Importer

The `wordpress` importer accepts one of:

- `options.xml`
- `options.buffer`
- `options.filePath` inside an import staging root

By default it returns a dry-run plan containing site metadata, authors,
WordPress terms, collection page plans, entries, attachments, comments
and skipped WordPress-only item types. Passing `dryRun: false` applies the plan
by emitting `registerContentType`, `createContentEntry`,
`createMediaAttachment` and `createComment`. When `pagesManager.createPage` is
available, WordPress categories are also projected as public collection pages
with `meta.isCollection: true`, and imported entries receive child page
projections under their primary category collection. Tags and additional terms
are preserved under each entry's `metadata.wordpress.terms` for migration
traceability; Blogposter collections are page parent/child structures, not
Content Engine taxonomies.
The importer also preserves conservative multilingual hints: recognized
WPML/Polylang-style language codes are forwarded as the Content Engine/Page
language, while translation group/source IDs remain under
`metadata.wordpress.translation`.
At render time, child page projections can inherit the nearest parent
collection's `meta.designId` or `meta.layoutTemplate`, so category-level design
choices cascade without recreating WordPress taxonomy templates.

Set `options.createPages: false` to disable the page/collection projection and
keep a content-only import.

The parser targets WordPress WXR exports, the XML format generated by WordPress' export tool.

## HTML Theme Importer

The `htmlTheme` importer accepts `options.sourceDir` pointing at a static HTML
theme folder inside an import staging root. Dry-run mode scans the folder,
detects HTML/CSS/image/font assets, extracts a page title and local asset
references, and returns an install plan.

Themes are presentation-only. The importer blocks installation when the source
contains JavaScript files, script tags, inline event handlers, executable source
files, remote references, CSS imports, remote CSS URLs or unsafe CSS behavior.
Those behaviors belong in widgets, modules or apps before the theme can be
installed.

Passing `dryRun: false` installs the theme under `public/themes/<slug>` by
copying the source files into `source/` and generating `theme.json` and
`theme.css`. Existing themes are not replaced unless `overwrite: true` is
passed.

## WordPress Site Package Importer

The `wordpressSitePackage` importer accepts a Blogposter WordPress site package
from one of:

- `options.packageDir`
- `options.sourceDir`
- `options.filePath` or `options.zipPath` pointing at a `.zip`
- `options.manifest` for tests or trusted internal calls

Package files must stay inside an import staging root when called through
`runImport`.

The package manifest must use one of these formats:

- `blogposter-wordpress-site-package`
- `wordpress-site-package`
- `wordpressSitePackage`

The importer keeps WordPress migration responsibilities split:

- `theme.styles` become candidate theme presentation assets.
- `pages[].rendered.htmlPath` or inline rendered HTML become page fallback
  content.
- `pages[].normalized.htmlPath` or inline normalized HTML become the
  Designer-mapping source while keeping rendered HTML as the visual fallback.
- `pages[].normalized.sourcePath` can point at WordPress source metadata such as
  post IDs, parent IDs, templates, builder detection, language/translation
  hints, terms, selected SEO data, featured media and sanitized post meta.
- `pages[].mapping.nativeWidgets`, `mapperHints`, `source` and `confidence`
  describe what can likely become native Blogposter Designer widgets.
- `content.wxr` can reuse the existing WordPress WXR importer.
- `assets[]` inventories every local package file captured from the rendered
  site, including CSS, JavaScript, images, icons, fonts and other static
  assets.
- `media[]` entries are applied through Media Manager events.
- `menus[]` entries are applied through Navigation Manager events when
  available. WordPress menu locations become Blogposter navigation locations,
  and menu item parent IDs are restored after the menu item replacement step
  when the manager returns inserted item IDs.
- `seo` site defaults and imported page paths are applied through SEO Manager
  events when available.
- `redirects[]` entries are applied through Redirect Manager events when
  available. Unsupported WordPress extension redirect actions stay in package
  metadata and warnings instead of being executed.
- `pages[].rendered.scripts` are reported as blocked behavior and are not
  converted into theme assets.
- `theme.scripts` blocks the package before writes begin.

Passing `dryRun: false` applies safe media and rendered page fallbacks through
existing core events. When normalized HTML is present, the Content Engine entry
stores a structured content payload with `html`, `normalizedHtml`,
`nativeWidgets`, `mapperHints` and `designerDraft` so the Design Studio can
rebuild from a cleaner source instead of scraping the rendered fallback again.
Visual package pages are ordered parent-before-child when source metadata
contains `parentSourceId`, so imported Blogposter pages can be created under
their WordPress parent and inherit presentation through the normal Pages
hierarchy.
Before writing content, the importer materializes local package assets from
`assets[]`, theme styles, page styles/scripts and media records through the
existing Media Manager `uploadFileToFolder` and `makeFilePublic` events when
that pipeline is available. Rendered HTML, normalized HTML, page style/script
metadata, media attachments and generated Designer drafts are then rewritten to
the resulting public Blogposter URLs. If the Media Manager pipeline is not
loaded, the importer leaves package-relative paths in place and records a
warning in the applied import result.
When the Navigation, SEO or Redirect managers are not loaded, those WordPress
records remain in the dry-run plan and applied warnings instead of creating
parallel importer-owned systems.
The dry-run plan also includes `styleHints` extracted from local theme/page CSS:
frequent colors, CSS variables, font families, spacing values and inferred token
roles. These hints are copied into `theme.tokens`, content metadata and the
generated Designer draft so Studio or an agent can rebuild the visual language
without treating WordPress CSS as executable theme behavior.
During write imports, each page also receives `behaviorHints` derived from its
rendered HTML, normalized HTML and page script list. These hints classify common
WordPress/Builder behavior such as sliders, animation libraries, forms, embeds,
maps and unknown custom scripts, then suggest rebuild targets like `gallery`,
`scene-effects`, `form-widget`, `embed-widget` or `review-widget-or-module`.
The hints are copied into content metadata and the generated Designer draft;
scripts remain blocked behavior and are not executed as theme code.
The draft maps recognized neutral HTML fragments to `textBox`, `mediaBlock`,
`buttonLink`, `navigationMenu` and `gallery` widget IDs, and keeps uncertain
markup as `htmlBlock` fallback widgets. If `designer.saveDesign` is available
during import, the importer also attempts to save the generated draft as an
editable Design Studio draft. Imported rendered pages are also projected into
Pages through `pagesManager.createPage` when available; the projection stores
the saved `designId` when one exists, keeps sanitized normalized/rendered HTML
as the fallback translation body, and passes `skipContentMirror: true` because
the canonical Content Engine entry was already created by the importer.
Otherwise the draft remains attached to the content entry for a later Studio
flow. JavaScript behavior still needs to be rebuilt as widgets, modules or apps.

## WordPress Visual Exporter

The WordPress-side prototype lives in
`tools/wordpress-visual-exporter/blogposter-visual-exporter.php`. It runs inside
WordPress, lets the active theme and builders render public pages, then exports a
neutral package:

- `pages/<slug>/rendered.html` keeps the captured frontend output.
- `pages/<slug>/normalized.html` removes scripts and volatile runtime
  attributes for Blogposter mapping.
- `pages/<slug>/source.json` records WordPress IDs, template, builder and
  meta-key hints plus parent, term, language, translation, SEO and selected
  sanitized post-meta data.
- `assets/` contains local CSS, JavaScript, images and fonts referenced by the
  captured pages.
- `manifest.json` records menus, the WordPress SEO summary and supported
  Redirection extension redirects for manager-level imports.
- `reports/mapping-hints.json`, `reports/blocked-behavior.json` and
  `reports/source-summary.json` make import confidence and review work explicit.

This exporter is a neutralizer, not an alternate runtime. It preserves the
rendered page and records JavaScript assets, while Blogposter imports the
normalized HTML and mapping hints for native rebuild work.
If WordPress cannot fetch its own frontend during export, the exporter writes a
WordPress-content fallback page and reports `BP_WP_EXPORT_RENDER_FALLBACK`
instead of aborting the whole site package.
Importer dry runs mirror exporter report warnings such as
`BP_WP_EXPORT_RENDER_FALLBACK` into `plan.warnings` and keep the original
warning objects under `plan.reports.exporterWarnings`.
Run `npm run package:wordpress-exporter` from the BlogposterCMS root to build
an installable `tools/wordpress-visual-exporter/dist/blogposter-visual-exporter.zip`
for WordPress' extension uploader.

## Visual Mapper

`mother/modules/importer/importers/wordpressVisualMapper.js` owns the first
neutral-HTML-to-Designer conversion pass. It is intentionally versioned and
heuristic because WordPress themes and builders do not share one structural
engine. The mapper:

- Preserves document order for recognized semantic fragments.
- Turns headings, paragraphs, lists and quotes into editable `textBox` widgets.
- Turns figures/images into `mediaBlock` widgets.
- Turns button-like anchors/buttons into `buttonLink` widgets.
- Turns navigation/gallery fragments into matching widget IDs with imported
  inline HTML as the first visual state.
- Drops unsafe imported `href`/`src` protocols before creating native widget
  draft markup.
- Uses `htmlBlock` only for unknown or custom fragments.

The mapper output is a draft, not a final perfect conversion. Review the
`summary.confidence`, `nativeWidgets` and `fallbackWidgets` values before
claiming an import is natively rebuilt.

`mother/modules/importer/importers/wordpressStyleHints.js` owns CSS hint
extraction for WordPress visual imports. It reads packaged local CSS only,
collects token candidates, and records warnings for unreadable style files. It
does not install a theme, execute CSS or allow JavaScript into the theme layer.

`mother/modules/importer/importers/wordpressBehaviorHints.js` owns script and
interaction hinting for WordPress visual imports. It classifies known library
names and rendered HTML markers without executing package JavaScript.
